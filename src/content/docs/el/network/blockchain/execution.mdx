Στο Aptos, η εκτέλεση αναφέρεται στη διαδικασία όπου οι validators εκτελούν και εφαρμόζουν συναλλαγές smart contracts
από ένα διατεταγμένο block. Η έξοδος εκτέλεσης στη συνέχεια εφαρμόζεται στην κατάσταση του blockchain.

Είναι κρίσιμο η εκτέλεση να είναι
[ντετερμινιστική](https://en.wikipedia.org/wiki/Deterministic_algorithm), και όλοι οι validators θα πρέπει
ιδανικά να συμφωνούν στην ίδια τελική κατάσταση του καθολικού. Μια πλειοψηφία των validators (περισσότερο από 2/3 των
validators) πρέπει να συμφωνεί στην τελική κατάσταση του καθολικού για να επιτευχθεί συναίνεση. Για περισσότερες
λεπτομερείς πληροφορίες σχετικά με τις ιδιότητες συναίνεσης, δείτε την ενότητα για το
[BFT](/el/network/glossary#byzantine-fault-tolerance-bft) και τη [συναίνεση](/el/network/glossary#consensus).

Η αποδοτική εκτέλεση, ιδιαίτερα η παράλληλη εκτέλεση, είναι κλειδί για την κλιμάκωση της απόδοσης και
της απόδοσης του blockchain. Η επίδραση αυτού μπορεί να γίνει αισθητή στον χρόνο block. Από τον Δεκέμβριο του 2024, τα blocks του Aptos κλείνουν μέσα σε 250ms.

## Γιατί Παράλληλη Εκτέλεση;

![](https://miro.medium.com/v2/resize:fit:4800/format:webp/0*qG0KRe1KzsP8Odth)

Η απλούστερη προσέγγιση για την εκτέλεση smart contracts είναι να εκτελούνται οι συναλλαγές σε ένα block
διαδοχικά, μία κάθε φορά. Ωστόσο, αυτό δεν κλιμακώνεται καλά. Το να έχεις μεγάλο αριθμό διαδοχικών
συναλλαγών, ειδικά με διαφορετικούς χρόνους εκτέλεσης, μπορεί να προκαλέσει δραστική πτώση της καθυστέρησης και της απόδοσης του
blockchain.

Για να λυθεί αυτό το πρόβλημα, τα blockchains άρχισαν να υιοθετούν παράλληλη εκτέλεση, ή την ικανότητα να επεξεργάζονται
πολλαπλές συναλλαγές παράλληλα. Η πρόκληση με την παράλληλη εκτέλεση, ωστόσο, είναι ότι ορισμένες
συναλλαγές μπορεί να διαβάζουν ή να γράφουν τον ίδιο πόρο, προκαλώντας συγκρούσεις.

### Στατικός Παραλληλισμός

Μια προσέγγιση για την αντιμετώπιση του θέματος συγκρούσεων στην παράλληλη εκτέλεση είναι ο **στατικός παραλληλισμός**, ο οποίος
απαιτεί από τους προγραμματιστές να καθορίσουν τις συγκρούσεις μεταξύ των συναλλαγών εκ των προτέρων. Αυτό, ωστόσο,
φέρνει μεγαλύτερη επιβάρυνση στον προγραμματιστή, και σε πολλές περιπτώσεις αναγκάζει συναλλαγές που δεν συγκρούονται πραγματικά
να είναι διαδοχικές.
