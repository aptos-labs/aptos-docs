Las redes locales pueden ser útiles cuando pruebas tu código. No están conectadas a ninguna red de producción de Aptos como mainnet, pero son útiles por tres razones principales:

1. **Sin límites de tasa:** Puedes interactuar con servicios alojados como la API de Nodo, API de Indexador y faucet sin límites de tasa para acelerar las pruebas.
2. **Reproducibilidad:** Puedes configurar escenarios específicos en la cadena y reiniciar la red desde cero en cualquier momento para volver a un estado limpio.
3. **Alta disponibilidad**: Las redes devnet y testnet de Aptos se actualizan periódicamente, durante las cuales pueden no estar disponibles. Las redes de desarrollo locales también están siempre disponibles incluso si no tienes acceso a internet.

<br />

# Iniciar una Red Local

<Steps>
  1. Asegúrate de tener la [CLI de Aptos](/build/cli) instalada.

  2. Asegúrate de tener [Docker](https://www.docker.com/) instalado.

     1. Esto es exclusivamente necesario para hacer un entorno similar a producción ejecutando la API de Indexador. Muchas herramientas posteriores como el SDK de Aptos dependen de la API de Indexador.
     2. Docker recomienda que instales vía [Docker Desktop](https://www.docker.com/products/docker-desktop/) para obtener actualizaciones automáticas.

  3. Inicia Docker.

  4. Ejecuta el siguiente comando en una nueva terminal para iniciar la red privada:

     ```shellscript filename="Terminal"
     aptos node run-local-testnet --with-indexer-api
     ```

     <Aside type="caution">
       Nota: A pesar del nombre (`local-testnet`), esto no tiene nada que ver con el testnet de Aptos, ejecutará una red completamente local en tu máquina.
     </Aside>

     Deberías esperar ver una salida similar a esta:

     ```shellscript filename="Terminal"

     Indexer API is starting, please wait...
     Node API is starting, please wait...
     ```

  5. Espera a que la red inicie

     Una vez que la terminal diga `Setup is complete, you can now use the local testnet!` la red local estará ejecutándose.

     <Aside type="caution">
       Si te encontraste con un error, mira los errores comunes a continuación para depurar.
     </Aside>

     <details>
       <summary>Errores Comunes al Iniciar la Red</summary>

       ### Dirección Ya en Uso

       ```shellscript filename="Terminal"
       panicked at 'error binding to 0.0.0.0:8080: error creating server listener: Address already in use (os error 48)'
       ```

       Esto significa que uno de los puertos necesarios para la red local ya está en uso por otro proceso.

       Para arreglar esto en sistemas Unix, puedes:

       1. Identificar el nombre y PID del proceso ejecutando `lsof -i :8080`.
       2. Ejecutar `kill <pid>` una vez que sepas el PID para liberar ese puerto.

       ### Error de demasiados archivos abiertos

       ```shellscript filename="Terminal"
       panicked at crates/aptos/src/node/local_testnet/logging.rs:64:10:
       called \`Result::unwrap()\` on an \`Err\` value: Os { code: 24, kind: Uncategorized, message: \"Too many open files\" }
       ```

       Esto significa que había demasiados archivos abiertos en tu sistema. En muchos sistemas Unix puedes aumentar el número máximo de archivos abiertos agregando algo como esto a tu `.zshrc`:

       ```shellscript filename="Terminal"
       ulimit -n 1048576
       ```

       ### Docker no está disponible

       ```shellscript filename="Terminal"
       Unexpected error: Failed to apply pre-run steps for Postgres: Docker is not available, confirm it is installed and running. On Linux you may need to use sudo
       ```

       Para depurar esto, prueba las siguientes correcciones:

       1. Asegúrate de tener docker instalado ejecutando `docker --version`.
       2. Asegúrate de que el daemon de Docker esté ejecutándose ejecutando `docker info` (si esto da error diciendo `Cannot connect to the Docker daemon` Docker NO está ejecutándose).
       3. Asegúrate de que el socket para conectar a Docker esté presente en tu máquina en la ubicación predeterminada. Por ejemplo, en sistemas Unix `/var/run/docker.sock` debería existir.
          1. Si ese archivo no existe, abre Docker Desktop y habilita `Settings -> Advanced -> Allow the default Docker socket to be used`.
          2. O, puedes encontrar dónde está el socket de Docker ejecutando `docker context inspect | grep Host`, luego crear un enlace simbólico de esa ubicación a la ubicación predeterminada ejecutando `sudo ln -s /Users/dport/.docker/run/docker.sock /var/run/docker.sock`
     </details>

     Como puedes ver desde la salida de ejemplo en el paso 4, una vez que la red local esté ejecutándose, tienes acceso a los siguientes servicios:

     - [API de Nodo](/build/apis/fullnode-rest-api): Esta es una API REST que se ejecuta directamente en el nodo. Permite funcionalidad de escritura central como envío de transacciones y un conjunto limitado de funcionalidad de lectura, como leer recursos de cuenta o información de módulos Move.
     - [API de Indexador](/build/indexer/indexer-api): Esta es una API [GraphQL](https://graphql.org/) que proporciona acceso de lectura rico a datos de blockchain indexados. Si haces clic en la URL para la API de Indexador arriba, por defecto [http://127.0.0.1:8090](http://127.0.0.1:8090/), abrirá la Consola de Hasura, una interfaz web que te ayudará a consultar la API GraphQL del Indexador.
     - [Servicio de Stream de Transacciones](/build/indexer/txn-stream): Este es un stream gRPC de transacciones usado por la API de Indexador y SDK. Esto solo es relevante para ti si estás desarrollando un procesador personalizado del [SDK de Indexador](/build/indexer/indexer-sdk).
     - [Postgres](https://www.postgresql.org/): Esta es la base de datos a la que escriben los procesadores del Indexador. La API de Indexador lee de esta base de datos.
     - [Faucet](/build/apis/faucet-api): Puedes usar esto para financiar cuentas en tu red local.

     Si no quieres ejecutar ninguno de estos subcomponentes de una red, hay banderas para deshabilitarlos.

     Si estás escribiendo un script y te gustaría esperar a que la red local se levante con todos los servicios, puedes hacer una solicitud GET a `http://127.0.0.1:8070`. Al principio esto devolverá código http [503](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/503). Cuando devuelva [200](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/200) significa que todos los servicios están listos.

     Para más información sobre diferentes banderas que puedes pasar al iniciar tu red local, o configuraciones como cambiar en qué puerto ciertos servicios se ejecutan, ejecuta el comando de ayuda:

     ```shellscript filename="Terminal"
     aptos node run-local-testnet --help
     ```
</Steps>

## Usar la Red Local

Ahora que la red está ejecutándose, puedes usarla como usarías cualquier otra red.

Entonces, puedes crear un perfil local así:

```shellscript filename="Terminal"
aptos init --profile <tu-nombre-de-perfil> --network local
```

Luego puedes usar ese perfil para cualquier comando que quieras usar en el futuro. Por ejemplo, si quisieras publicar un módulo Move como el paquete [`hello_blockchain`](https://github.com/aptos-labs/aptos-core/tree/main/aptos-move/move-examples/hello_blockchain) a tu red local podrías ejecutar:

```shellscript filename="Terminal"
aptos move publish --profile <tu-nombre-de-perfil> --package-dir /opt/git/aptos-core/aptos-move/move-examples/hello_blockchain --named-addresses HelloBlockchain=local
```

### Configurar el SDK de TypeScript

Si quieres usar la red local con el SDK de TypeScript, puedes usar URLs de red local al inicializar el objeto cliente (`Aptos`):

```tsx
import { Aptos, AptosConfig, Network } from "@aptos-labs/ts-sdk";

const network = Network.LOCAL;
const config = new AptosConfig({ network });
const client = new Aptos(config);
```

### Reiniciar la red local

A veces mientras desarrollas es útil reiniciar la red local de vuelta a su estado inicial, por ejemplo:

- Hiciste cambios incompatibles hacia atrás a un módulo Move, y te gustaría redeployarlo sin renombrarlo o usar una nueva cuenta.
- Estás construyendo un procesador personalizado del [SDK de Indexador](/build/indexer/indexer-sdk) y te gustaría indexar usando una red fresca.
- Quieres limpiar todo el estado en la cadena, e.g., cuentas, objetos, etc.

Para comenzar con una red local completamente nueva, usa la bandera `--force-restart`:

```shellscript filename="Terminal"
aptos node run-local-testnet --force-restart
```

Luego te preguntará si realmente quieres reiniciar la cadena, para asegurar que no elimines tu trabajo por accidente.

```shellscript filename="Terminal"
Are you sure you want to delete the existing chain? [yes/no]
> yes
```

Si no quieres ser preguntado, incluye `--assume-yes` también:

```shellscript filename="Terminal"
aptos node run-local-testnet --force-restart --assume-yes
```
