Para asegurar que tu public fullnode (PFN) opere sin problemas, debería cumplir los requisitos especificados en este documento.

## Requisitos de hardware

<Aside type="caution">
  **Requisitos mínimos de hardware**<br />
  Fallar en cumplir los requisitos mínimos de hardware significa que tu PFN experimentará degradación bajo
  carga e inestabilidad general en entornos de producción. No se recomienda ejecutar un PFN sin
  cumplir los requisitos mínimos de hardware.
</Aside>

Para ejecutar un PFN de grado de producción, recomendamos que tu hardware cumpla los mismos requisitos que un
validator o VFN. Puedes ver los requisitos de hardware para estos, aquí: [requisitos de hardware de validator y VFN](/network/nodes/validator-node/node-requirements#hardware-requirements).

<Aside type="note">
  **PFNs de prueba**<br />
  Si deseas ejecutar un PFN para desarrollo o pruebas solamente, se pueden usar especificaciones de hardware menores.
  Pero, no debería usarse en producción.
</Aside>

## Requisitos de red y puertos

Cuando estés ejecutando un PFN, se te requiere abrir puertos de red en tus nodos para permitir que otros nodos (es decir, pares)
se conecten a ti. Hay diferentes tipos de redes Aptos, y cada tipo de red usa un puerto diferente. Sin embargo,
el único tipo de red que un PFN usa es la red pública, donde los PFNs se conectan a otros PFNs y VFNs.

Tu PFN puede configurarse para que la red pública opere usando un puerto específico en tu nodo. Puedes configurar
las configuraciones de puerto usando el archivo YAML de configuración del nodo. Aquí hay un [archivo de
configuración de ejemplo](https://github.com/aptos-labs/aptos-core/blob/main/config/src/config/test_data/public_full_node.yaml#L16) para un PFN
que configura la red pública para usar el puerto `6180`.

### Configuraciones de puerto

Las recomendaciones descritas abajo asumen las configuraciones de puerto por defecto usadas por PFNs. Si has
cambiado las configuraciones de puerto por defecto en tu archivo de configuración, entonces deberías ajustar las recomendaciones en consecuencia.

<Aside type="caution">
  **Exponiendo puertos**<br />
  A menos que se requiera explícitamente, recomendamos que no expongas ningún otro puerto mientras operas un PFN. Esto es porque
  exponer puertos adicionales puede aumentar la superficie de ataque de tu nodo y hacerlo más vulnerable a adversarios.
</Aside>

#### Ejecutando un PFN:

Asumiendo que se usan puertos por defecto, lo siguiente debería configurarse para PFNs:

- Abrir los siguientes puertos TCP:
  - `6182` – **Red pública**: Abre este puerto públicamente para habilitar a otros PFNs a conectarse a tu PFN.
- Cerrar los siguientes puertos TCP:
  - `9101` – **Servicio de inspección**: Cierra este puerto para prevenir inspección de métricas no autorizada.
  - `9102` – **Servicio admin**: Cierra este puerto para prevenir interacción de servicio admin no autorizada.
  - `80/8080` - **API REST**: Cierra este puerto para prevenir acceso a API REST no autorizado.

<Aside type="caution">
  **Exponiendo servicios**<br />
  El puerto del servicio de inspección (`9101`), puerto del servicio admin (`9102`) y el puerto de la API REST (`80` o `8080`)
  son probablemente útiles para tu red interna, ej., desarrollo de aplicaciones y debugging. Sin embargo, el puerto del servicio de inspección
  y el puerto del servicio admin nunca deberían exponerse públicamente ya que pueden ser fácilmente abusados. Similarmente, si eliges
  exponer el endpoint de la API REST públicamente, deberías desplegar un mecanismo adicional de autenticación o limitación de tasa para
  prevenir abuso.
</Aside>

## Requisitos de almacenamiento

La cantidad de datos almacenados por PFNs de Aptos depende del historial del ledger (longitud) de la blockchain y el número de
estados on-chain (ej., cuentas y recursos). Tanto el historial del ledger como el número de
estados on-chain dependen de varios factores adicionales, incluyendo la edad de la blockchain, la tasa promedio de transacciones
a lo largo del tiempo, y la configuración del podador de base de datos del ledger. Al momento de escribir, estimamos
que los PFNs de testnet y mainnet requieren varios 100's de GB de almacenamiento.

Nota que porque los nodos archivales almacenan toda la historia de la blockchain, el tamaño de la base de datos en nodos archivales
continuará creciendo sin límites. Como resultado, no podemos proporcionar una recomendación para tamaños de almacenamiento de nodos archivales.

<Aside type="note">
  **Almacenamiento de blockchain devnet**<br />
  La devnet de Aptos actualmente se reinicia semanalmente. Si estás desplegando un PFN de devnet, esto significa que el almacenamiento
  se reiniciará (es decir, se limpiará) cada semana. Ver el canal `#devnet-release` en el [Discord de Aptos](https://discord.gg/aptosnetwork).
</Aside>
