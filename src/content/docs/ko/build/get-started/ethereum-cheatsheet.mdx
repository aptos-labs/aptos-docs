### 높은 수준의 개요

| 기능                        | Ethereum                                                                                                                       | Aptos                                                                 |
|-----------------------------|-------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------|
| **스마트 컨트랙트**        | Solidity, EVM                                                                                                                  | Move, MoveVM                                                          |
| **장점**                    | 성숙, 광범위한 채택                                                                                                           | 확장성, 낮은 지연 시간, 예측 가능한 수수료                             |
| **트랜잭션 수수료**        | 변수, 높을 수 있음                                                                                                            | 더 낮고 더 예측 가능                                                 |
| **계정 주소**              | 160비트                                                                                                                        | 256비트                                                               |
| **계정 구조**              | 단일 필드의 잔액, nonce 사용                                                                                                   | 모듈 및 리소스, 시퀀스 번호 사용                                     |
| **데이터 저장**            | Patricia Merkle 트리                                                                                                          | 리소스 및 모듈이 있는 글로벌 저장                                    |
| **저장 정신**              | 계약 기반 저장                                                                                                                 | 코드 및 데이터에 대한 계정 중심 정신                                 |
| **예제 코드**              | [ERC-20](https://github.com/OpenZeppelin/openzeppelin-contracts/tree/master/contracts/token/ERC20)                             | [Fungible Asset](https://github.com/aptos-labs/aptos-core/blob/main/aptos-move/framework/aptos-framework/sources/fungible_asset.move) |
| **호출자 ID**              | `msg.sender`                                                                                                                   | `&signer` 참조                                                        |
| **업그레이드 가능성**      | 프록시 패턴                                                                                                                    | 직접 모듈 업그레이드                                                  |
| **안전성 및 보안**         | 재진입과 같은 공격에 취약                                                                                                      | 일반적인 취약점 완화                                                  |
| **디스패치 유형**          | 동적 디스패치                                                                                                                  | 정적 디스패치                                                         |
| **FT 표준**                | [ERC-20](https://docs.openzeppelin.com/contracts/4.x/erc20)                                                                    | [Coin](/build/smart-contracts/aptos-coin) (legacy) 및 [Fungible Asset](/build/smart-contracts/fungible-asset) |
| **NFT 표준**               | [ERC-721](https://docs.openzeppelin.com/contracts/4.x/erc721), [ERC-1155](https://docs.openzeppelin.com/contracts/4.x/erc1155) | [Digital Asset](/build/smart-contracts/digital-asset)                 |
| **블록체인 상호작용**      | [Ethers.js 라이브러리](https://docs.ethers.org/v6/)                                                                            | [Aptos TypeScript SDK](/build/sdks/ts-sdk)                            |

### 토큰 표준의 세부 비교

|                        | Solidity                                                               | Move (Aptos)                                                                 |
|------------------------|------------------------------------------------------------------------|------------------------------------------------------------------------------|
| **토큰 구조**         | 각 토큰은 자체 계약입니다.                                            | 모든 토큰은 단일 재사용 가능한 계약을 사용하여 유형화된 `Coin` 또는 `FungibleAsset`입니다. |
| **토큰 표준**         | ERC20과 같은 표준을 준수해야 함; 구현이 다를 수 있음.                    | 모든 토큰에 대한 균일한 인터페이스 및 구현.                                 |
| **잔액 저장**         | 매핑 구조를 사용하여 계약에 잔액 저장.                                | **리소스 지향 잔액**: 잔액이 사용자의 계정에 리소스로 저장됩니다. 리소스는 임의로 생성될 수 없어 토큰 가치의 무결성을 보장합니다. |
| **전송 메커니즘**     | 수령인의 명시적 허가 없이 토큰을 전송할 수 있음.                      | 특정 경우(예: AptosCoin)를 제외하고, 토큰은 일반적으로 전송을 위해 수령인의 `signer` 권한이 필요합니다. |

### EVM과 Move VM의 세부 비교

- **EVM**: 유연성과 동적 디스패치로 유명하며, 이는 광범위한 스마트 컨트랙트 동작을 허용합니다. 이 유연성은 그러나 병렬 실행 및 네트워크 작업에서 복잡성을 초래할 수 있습니다.
- **Move VM**: VM과 프로그래밍 언어 사이의 보다 통합된 접근 방식으로 보안과 효율성에 초점을 맞춥니다. 데이터 저장 모델은 더 나은 병렬화를 허용하며, 정적 디스패치 방법은 보안과 예측 가능성을 향상시킵니다.

<br />

|                                 | EVM (Ethereum Virtual Machine)                                         | Move VM (Move Virtual Machine)                                     |
|---------------------------------|------------------------------------------------------------------------|-------------------------------------------------------------------|
| **데이터 저장**                | 데이터가 스마트 컨트랙트의 저장 공간에 저장됩니다.                    | 데이터가 스마트 컨트랙트, 사용자 계정 및 객체에 걸쳐 저장됩니다.    |
| **병렬화**                     | 공유 저장 공간으로 인해 병렬 실행이 제한됩니다.                       | 유연한 분할 저장 설계로 인해 더 많은 병렬 실행이 가능합니다.        |
| **VM 및 언어 통합**            | EVM과 스마트 컨트랙트 언어(예: Solidity)에 대한 별도의 계층.          | VM 계층과 Move 언어 사이의 원활한 통합, Rust에서 작성된 기본 함수가 Move에서 실행 가능. |
| **중요 네트워크 작업**         | 네트워크 작업 구현이 복잡하고 덜 직접적일 수 있습니다.                 | 검증자 세트 관리와 같은 중요한 작업이 Move에서 기본적으로 구현되어 직접 실행을 허용합니다. |
| **함수 호출**                  | 동적 디스패치가 임의의 스마트 컨트랙트 호출을 허용합니다.             | 정적 디스패치가 보안과 예측 가능한 동작에 초점을 맞춥니다.         |
| **유형 안전성**                | 계약 유형이 유형 안전성의 수준을 제공합니다.                          | Move의 모듈 구조 및 제네릭이 강력한 유형 안전성을 제공합니다.       |
| **트랜잭션 안전성**            | 트랜잭션 순서 및 안전성을 위해 nonce를 사용합니다.                    | 트랜잭션 순서 및 안전성을 위해 시퀀스 번호를 사용합니다.           |
| **인증된 저장**                | 스마트 컨트랙트 저장으로 예.                                        | Move의 리소스 모델을 활용하여 예.                                 |
| **객체 접근성**                | 객체가 전역적으로 접근할 수 없음; 스마트 컨트랙트 범위에 바인딩됨.    | 객체의 전역 접근성이 보장됩니다.                                   |
